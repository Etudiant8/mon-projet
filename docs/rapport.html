01 Contexte et objectifs

Rappel du besoin métier et de ce que la chaîne devait démontrer.

L'entreprise fictive Catal-Log souhaite industrialiser la publication d'un site web statique afin d'éviter les opérations manuelles répétitives et de conserver des preuves d'exécution fiables. L'objectif de ce projet était donc de mettre en place une chaîne CI/CD simple, lisible et traçable, capable de :

  • construire une image Docker Nginx à partir d'un Dockerfile ;
  • tester automatiquement cette image dans GitHub Actions ;
  • publier l'image dans GitHub Container Registry (GHCR) avec un tag et un digest identifiables ;
  • valider l'image dans un environnement de recette simulé ;
  • promouvoir manuellement le même artefact vers une production simulée, sans reconstruction.

Le travail a été mené sur un dépôt individuel déjà existant et partiellement avancé, plutôt que sur le modèle de projet vierge fourni initialement.

02 Méthodologie de travail

Comment le travail a été organisé, du diagnostic à la validation.

La démarche a suivi une logique d'audit avant action : commencer par comprendre l'existant avant de modifier quoi que ce soit, puis avancer par petites itérations vérifiées à chaque étape plutôt que par une refonte unique.

1
Lecture et analyse

Lecture complète du sujet EC06 (PDF de consignes) puis inspection de tous les fichiers du dépôt modèle : workflows, Dockerfile, compose.yml, site, documentation.

2
Vérification des accès avant toute action

Confirmation explicite de l'accès réel au compte GitHub et à Docker avant de commencer à modifier quoi que ce soit, pour éviter de travailler sur de mauvaises hypothèses.

3
Tests avant modification

Chaque changement (site, Dockerfile, documentation) a été suivi d'un test réel : docker build, docker run, docker compose up, avant de considérer l'étape terminée.

4
Validation par preuve, pas par déclaration

Les valeurs inscrites dans la documentation (tags, digests, liens de run) ont été récupérées directement depuis GHCR et l'API GitHub Actions après exécution réelle, plutôt qu'estimées à l'avance.

03 État initial du dépôt

Le point de départ réel du projet, différent du modèle vierge.

Le dépôt mon-projet n'était pas un squelette vide : il contenait déjà un historique de commits, des workflows fonctionnels, un site personnalisé et une documentation partiellement rédigée, avec des preuves (digests, liens) correspondant à une version antérieure du site. Ce constat a orienté la suite du travail vers de la consolidation et une mise à jour cohérente, plutôt que vers une reconstruction complète.

04 Vérification des accès

Aucune supposition : chaque accès a été testé avant d'être utilisé.

Avant toute intervention, l'accès à GitHub et à Docker a été vérifié concrètement plutôt que supposé :

  • GitHub : connexion confirmée au compte, identification du dépôt réel du projet parmi les dépôts existants.
  • Docker : Docker Desktop détecté comme installé mais avec le démon arrêté ; démarrage explicite puis vérification par docker info avant utilisation.
Pourquoi c'est important : partir d'un état vérifié évite de construire une documentation ou une démonstration sur une hypothèse fausse (par exemple, croire que Docker est disponible alors que le service n'est pas démarré).

05 Tests techniques locaux

Chaque brique a été testée isolément avant d'être considérée acquise.

Build et exécution du conteneur

docker build -t projet-cicd-nginx:local .
docker run -d --name site-test -p 8080:80 projet-cicd-nginx:local
curl -fsS http://127.0.0.1:8080/
curl -fsS http://127.0.0.1:8080/version.json

Résultat : réponse HTTP 200 sur les deux endpoints, healthcheck du conteneur passé à l'état healthy.

Orchestration avec Docker Compose

docker compose up --build

Résultat : le service web démarre et passe à l'état healthy ; le service tester exécute ses vérifications HTTP et se termine avec un code de sortie 0, preuve que la coordination entre les deux conteneurs fonctionne.

06 Anonymisation du projet

Suppression de toute trace du nom réel, avec mesure des conséquences avant action.

La demande d'anonymisation touchait deux niveaux différents, traités séparément :

Niveau texte

Remplacement de toutes les occurrences du nom d'auteur (« Anonyme ») par Etudiant_8 dans le site, le Dockerfile, compose.yml et l'ensemble des fichiers docs/*.md.

Niveau compte GitHub

Le pseudo GitHub DEimazoute apparaissait dans toutes les URL du projet (dépôt, GHCR). Avant de le renommer, les conséquences ont été vérifiées et présentées explicitement : GitHub prévient qu'un changement de pseudo ne redirige pas l'ancienne page de profil ni les sites GitHub Pages, contrairement aux dépôts qui sont redirigés automatiquement. Le renommage vers Etudiant8 n'a été effectué qu'après validation explicite de ce risque.

Point de vigilance documenté : un renommage de compte GitHub est une action à l'échelle du compte entier, pas seulement du dépôt concerné — tous les autres dépôts personnels sont également affectés.

08 Publication et promotion

Le passage du local vers le réel : commit, publication automatique, validation manuelle.

Une fois les changements validés en local, ils ont été commités et poussés sur la branche main, déclenchant automatiquement 01-ci.yml et 02-publish-ghcr.yml. La promotion vers production-simulee a ensuite été déclenchée manuellement via workflow_dispatch.

Étape inattendue : l'environnement GitHub production-simulee est protégé par une règle de validation humaine obligatoire. Le workflow s'est mis en attente (waiting) jusqu'à approbation manuelle explicite dans l'interface GitHub — un mécanisme de sécurité qui correspond exactement à ce que le sujet demande d'expliquer en section sécurité.

La preuve que la promotion s'est faite sans reconstruction repose sur la comparaison des digests : l'image source (latest) et l'image cible (production-simulee) partagent rigoureusement le même digest SHA-256, confirmé après docker pull des deux tags.

09 Difficultés rencontrées

Ce qui ne s'est pas passé du premier coup, et comment ça a été résolu.

  • Docker Desktop non démarré : le démon n'était pas actif au premier essai ; démarrage explicite puis nouvelle vérification avant de continuer.
  • Digest instable entre deux builds identiques : chaque push relance 02-publish-ghcr.yml, y compris pour un changement de documentation seul. docker/build-push-action attache des métadonnées de provenance horodatées, ce qui change le digest même quand le site ne change pas. Il a fallu resynchroniser les preuves après le dernier push réel plutôt que de les figer trop tôt.
  • Environnement protégé en attente de validation : la promotion vers production-simulee nécessitait une approbation manuelle, non automatisable — traité comme une étape normale du processus plutôt qu'une erreur.
  • Connexion à l'outil de navigateur instable : quelques tentatives ont été nécessaires avant de pouvoir piloter l'interface GitHub pour déclencher et approuver les workflows.

11 Bilan

Ce que cette démarche démontre au-delà du résultat final.

Au-delà de la chaîne CI/CD elle-même, ce travail a nécessité de vérifier chaque hypothèse avant d'agir (accès, disponibilité de Docker, contenu réel du dépôt), de tester chaque modification avant de la considérer acquise, et de traiter les imprévus (règle de validation manuelle, instabilité du digest) comme des éléments à comprendre et documenter plutôt qu'à contourner. C'est cette rigueur, plus que le résultat final seul, qui permet de expliquer et défendre la chaîne mise en place.
